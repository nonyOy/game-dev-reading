# 2026-07-07：游戏 UI 从美术稿到引擎落地的制作流程

上一篇整理了 AI 生图做游戏 UI 时最容易踩的坑：AI 很容易生成一张“看起来像 UI 的完整图”，但游戏真正需要的是可拆分、可拉伸、可换状态、可被程序控制的资源。

今天继续往后写一步：当我已经有了 UI 美术稿，怎么把它变成引擎里真正能用的东西。

我觉得这是现在最容易被忽略的一步。很多 UI 看起来不够出品，不一定是因为设计差，也可能是因为它没有经过“工程落地”的整理。概念图里很漂亮的按钮、血条、面板、图标，放进游戏以后可能会出现这些问题：

- 缩放以后边框变糊。
- 九宫格切错，角被拉变形。
- 字体压住装饰。
- 长文本放不下。
- 选中态不明显。
- 血条从 100% 变到 5% 时穿帮。
- UI 在 16:9 正常，在 19.5:9 手机上错位。
- PC 上看着正好，手机上字太小。
- 动效太慢，影响操作节奏。
- 资源命名混乱，后面没人敢改。

所以 UI 美术的完成，不是 PSD 里画完，也不是 AI 出图完，而是要到引擎里跑起来以后才算真正接近完成。

这篇就专门整理一个流程：从美术稿到引擎落地，应该怎么拆、怎么切、怎么命名、怎么适配、怎么验收。

## 一、先把“展示图”和“游戏资源”分开

做 UI 时，最先要分清两种东西：

```text
展示图：给人看的效果图，用来表达风格、构图和气质。
游戏资源：给引擎用的零件，要能被程序控制和复用。
```

展示图可以很完整：

```text
一个完整战斗 HUD
一张带血条、头像、技能按钮、伤害数字的界面
一套背包界面概念图
一个角色详情页视觉稿
```

但游戏资源不能这样交付。游戏资源要拆成：

```text
背景
边框
按钮
按钮状态
图标
文字样式
遮罩
填充条
高光
特效
九宫格面板
动效片段
```

这一步的核心是：展示图负责说明“长什么样”，游戏资源负责说明“怎么工作”。

如果直接把展示图切成一整张图片放进引擎，短期看起来很快，后期一定会痛苦。因为任何变化都会变成返图：

```text
血量变了，要返图。
按钮文字变了，要返图。
面板内容变长，要返图。
换语言，要返图。
加选中态，要返图。
改尺寸，要返图。
```

真正的游戏 UI 要避免这种死图依赖。

## 二、进入引擎前，先做 UI 资源拆分表

不要直接打开引擎开始摆。先写一张资源拆分表。

比如一个战斗主界面，可以拆成这样：

```text
HUD_PlayerStatus
  ui_avatar_frame.png
  ui_avatar_mask.png
  ui_hp_frame.png
  ui_hp_empty.png
  ui_hp_fill.png
  ui_hp_damage_delay.png
  ui_mp_frame.png
  ui_mp_empty.png
  ui_mp_fill.png
  ui_status_icon_poison.png
  ui_status_icon_shield.png

HUD_SkillButtons
  ui_skill_slot_normal.png
  ui_skill_slot_selected.png
  ui_skill_slot_disabled.png
  ui_skill_cooldown_mask.png
  ui_skill_key_hint_bg.png

HUD_BossStatus
  ui_boss_hp_frame.png
  ui_boss_hp_empty.png
  ui_boss_hp_fill.png
  ui_boss_nameplate.png
```

这张表的意义很大。它能提前暴露问题：

- 哪些资源其实可以复用？
- 哪些状态还没画？
- 哪些文字不能烘焙进图片？
- 哪些地方需要遮罩？
- 哪些资源需要九宫格？
- 哪些资源需要程序动态换颜色？
- 哪些图标需要统一尺寸？

没有拆分表，UI 很容易变成“边做边补”。最后工程里会出现一堆名字混乱的图片，比如：

```text
button.png
button2.png
button_new_final.png
button_final2.png
hpbar_改.png
hpbar_最终.png
ui1.png
```

这种资源一多，项目就很难维护。

## 三、九宫格不是可选项，而是面板和按钮的基本能力

很多游戏 UI 面板都需要适配不同内容长度。比如任务描述、背包说明、弹窗、技能详情、设置面板。它们不可能每种尺寸都画一张图。

这就需要九宫格。

九宫格的意思是把一张 UI 图分成九块：

```text
左上角    上边    右上角
左边      中间    右边
左下角    下边    右下角
```

其中四个角通常保持不拉伸，边可以横向或纵向拉伸，中间区域可以拉伸或平铺。

适合九宫格的资源：

```text
按钮背景
弹窗面板
提示框
任务框
背包格子
输入框
标签页
商店商品卡片
```

不适合直接九宫格的资源：

```text
复杂插画背景
透视很强的装饰
中心有固定图案的图
边框纹理不连续的图
左右极不对称的造型
有大面积斜切且不能分段的图
```

如果一个按钮有很复杂的左右装饰，直接九宫格可能会把中间拉得很丑。这时可以改成三段式：

```text
button_left_cap.png
button_center_tile.png
button_right_cap.png
```

血条也常用三段式：

```text
hp_left_cap.png
hp_center.png
hp_right_cap.png
```

这样左右端头不变，中间区域拉伸或平铺。

我以后做 UI 时，要养成一个习惯：画面板和按钮时，先想九宫格线在哪里，再决定装饰放哪里。

如果装饰刚好压在拉伸区域，后面一定会出问题。

## 四、血条进引擎后真正要测的是变化过程

血条在静态图里很容易好看，但进引擎后要测的是 0% 到 100% 的变化。

一个合格血条至少要测试这些状态：

```text
100%：满血时是否完整。
75%：填充是否自然。
50%：中段是否能正常裁切。
25%：低血量是否仍然清楚。
5%：极低血量时是否穿帮。
0%：空血槽是否好看。
受伤瞬间：延迟条是否和主血条区分明确。
回血：填充增长是否平滑。
中毒：颜色变化是否还能读。
护盾：额外层是否压住血量信息。
低血量闪烁：警示是否明显但不过度刺激。
```

如果只看 100% 满血，很容易误判。

血条常见实现方式：

```text
横向裁切：用 Mask 或 Filled Image 控制填充宽度。
横向缩放：直接缩放 fill 的 X 轴。
材质遮罩：用 shader 或 mask 控制不规则填充。
分段显示：用多个小格子表示生命段。
叠层显示：主血条、延迟条、护盾条分层。
```

横向缩放最简单，但容易让纹理变形。横向裁切更稳定。如果血条填充有液体纹理、高光、斜纹，最好用遮罩裁切，而不是把图片直接压扁。

血条外框不要跟填充合在一起。外框是静态层，填充是动态层。高光如果需要随血量减少，也应该跟填充一起被裁切。

一个简单层级可以是：

```text
HPBar
  EmptySlot
  DamageDelayFill
  FillMask
    Fill
    FillHighlight
  Frame
  LowHpGlow
  ValueText
```

这个层级比一张整图麻烦，但它才是可控的。

## 五、按钮至少要做四个状态

很多 UI 看起来不出品，是因为只有普通状态，没有交互状态。

按钮最少要有：

```text
Normal：默认状态。
Selected / Hover：被选中或鼠标悬停。
Pressed：按下瞬间。
Disabled：不可点击。
```

如果是手游，还可能需要：

```text
New：有新内容。
Locked：未解锁。
Cooldown：冷却中。
Equipped：已装备。
RedDot：红点提示。
```

按钮状态不是简单调亮调暗。要保证玩家在复杂背景里能一眼看出来：

```text
我当前选中了哪个？
这个能不能点？
我刚才有没有点成功？
这个功能是否有新内容？
```

选中态可以通过这些方式做：

```text
外框发光
底色变化
边缘描边增强
轻微放大
装饰展开
箭头或焦点框
音效和动效配合
```

禁用态要避免只是透明度降低。透明度太低会让按钮和背景混在一起。更好的做法是：

```text
降低饱和度
减少高光
保留轮廓
加锁图标
禁用文字变灰
取消动效
```

按钮是否出品，很大程度看状态是否完整。

## 六、字体不是最后随便放上去的

UI 美术里，字体经常被低估。

很多界面在美术稿里很好看，但一进引擎就崩，是因为字体没有提前规划。常见问题：

- 中文比英文占空间大。
- 英文单词太长，按钮放不下。
- 数字位数变化导致排版跳动。
- 字体描边太粗，小字糊成一团。
- 字号太多，界面层级混乱。
- 标题字很漂亮，正文难读。
- 发光和阴影压住文字。

所以 UI 落地前要做字体规范。

至少定义这些：

```text
标题字体
正文字体
数字字体
按钮字体
伤害数字字体
提示字体
禁用态颜色
强调色
描边规则
阴影规则
最小字号
```

还要提前测试真实文本。

不要只放：

```text
开始
确定
攻击
```

还要测试：

```text
自动战斗
领取全部奖励
无法在战斗中更换装备
Super Critical Damage
Inventory Capacity Full
```

如果 UI 要支持多语言，按钮和面板必须给文本留弹性空间。不要让装饰把文字区域卡死。

## 七、适配不是简单等比缩放

游戏 UI 必须面对不同分辨率和屏幕比例。

常见比例：

```text
16:9：PC 和主机常见。
16:10：部分显示器和掌机。
21:9：超宽屏。
4:3：部分旧设备或特殊窗口。
19.5:9：手机常见。
```

如果只在 1920x1080 下摆好，换屏幕就可能出问题。

UI 适配要考虑：

```text
锚点：元素跟随左上、右上、底部、中间还是某个安全区。
安全区：手机刘海屏、圆角屏、系统手势区域。
缩放规则：按宽缩放、按高缩放，还是按最小边缩放。
布局规则：横屏、竖屏是否不同。
边距规则：不同分辨率下是否保持视觉呼吸。
字体规则：小屏是否增大字号或减少信息。
```

HUD 元素通常不要全部固定在画布中心。比如：

```text
玩家血条锚到左上。
小地图锚到右上。
技能按钮锚到底部或右下。
任务提示锚到左侧。
弹窗居中。
```

但锚点只是第一步。还要测极端情况：

```text
超宽屏时，HUD 是否离中心太远？
手机窄屏时，技能按钮是否挤在一起？
弹窗是否超出安全区？
长文本是否被遮挡？
```

适配做不好，UI 美术再好也会显得不专业。

## 八、动效要服务操作，不要只服务炫

UI 动效很重要，但动效不是越多越好。

动效的任务有几个：

```text
确认操作：玩家知道点中了。
引导视线：让玩家看到新内容。
表达层级：重要信息出现得更有重量。
制造情绪：胜利、失败、升级、获得奖励更有感觉。
隐藏切换：让界面变化不突兀。
```

但动效也会带来问题：

```text
太慢会拖操作节奏。
太花会干扰读信息。
太频繁会疲劳。
太大幅度会让玩家找不到原位置。
不可跳过会烦。
```

所以动效要分等级。

高频操作：

```text
按钮按下
标签切换
背包选择
技能释放
```

这些动效要短，通常 0.08 到 0.2 秒就够了。

中频反馈：

```text
打开面板
关闭面板
获得物品
任务更新
```

可以稍微完整一点。

低频演出：

```text
升级
抽卡结果
Boss 击败
章节完成
胜利结算
```

可以更华丽。

一个常见错误是把所有 UI 都做成低频演出。第一次看很好，玩久了会烦。

## 九、资源命名和目录结构决定后期能不能维护

UI 资源一多，命名就变得非常重要。

命名要让人一眼看懂：

```text
ui_hp_frame.png
ui_hp_fill_red.png
ui_hp_fill_poison.png
ui_btn_primary_normal.png
ui_btn_primary_selected.png
ui_btn_primary_pressed.png
ui_btn_primary_disabled.png
ui_panel_popup_bg.png
ui_icon_item_potion.png
ui_icon_status_poison.png
```

不要用：

```text
新建图层 1.png
按钮最终.png
按钮最终改.png
图标2.png
血条新.png
UI_001.png
```

目录也要清楚：

```text
UI/
  Common/
    Buttons/
    Panels/
    Icons/
    Fonts/
  HUD/
    Player/
    Enemy/
    Skills/
  Inventory/
  Battle/
  Shop/
  Effects/
```

如果项目小，也不需要过度复杂。但至少要保证一个原则：别人看到文件名，不打开图也大概知道它是什么。

## 十、UI 验收要在真实场景里做

UI 美术稿通常是在干净背景上看的。但游戏里背景很复杂。

所以必须在真实场景里验收。

要测试这些背景：

```text
明亮背景
黑暗背景
高饱和背景
高运动背景
战斗特效很多的时候
角色和怪物遮挡的时候
镜头快速移动的时候
```

很多 UI 在灰色画布上很清楚，放到游戏里就被背景吃掉。这时要考虑：

```text
加暗底
加强描边
降低透明度
增加背景模糊
增加局部遮罩
固定 HUD 区域
调整颜色对比
```

UI 不是独立画面，它永远和游戏画面叠在一起。

## 十一、一个完整的 UI 落地流程

以后做 UI，可以按这个流程走：

```text
1. 功能需求
   明确这个界面解决什么问题，有哪些操作和状态。

2. 信息层级
   先排主信息、次信息、操作按钮、反馈区域。

3. 风格稿
   确定视觉方向、形状语言、色彩、字体、装饰。

4. 资源拆分表
   列出需要导出的图片、字体、动效、状态。

5. 源文件分层
   PSD/Figma 中按组件整理，不把功能层粘死。

6. 九宫格和切图
   面板、按钮、槽位、弹窗都按适配需求切。

7. 引擎搭建
   用真实 UI 组件摆出来，不用一张整图糊上去。

8. 状态接入
   普通、选中、按下、禁用、冷却、红点、数值变化。

9. 分辨率适配
   测 PC、手机、宽屏、小屏、安全区。

10. 真实场景验收
   放到战斗、背包、商店、地图等实际画面中看。

11. 交互和动效调整
   确认操作反馈、节奏、可跳过性和疲劳感。

12. 回修资源
   根据引擎表现回到美术文件修图、补状态、改切图。
```

这套流程比“画完导出”麻烦，但它能避免 UI 卡在概念阶段。

## 十二、出品级 UI 的最低检查表

以后做一个 UI 组件，至少要检查这些：

```text
它是否有普通、选中、按下、禁用状态？
它是否能在真实尺寸下看清？
它是否能适配长文本？
它是否能九宫格或三段式拉伸？
它是否有清楚的命名？
它是否和同类组件风格一致？
它是否能在复杂背景上读清？
它是否需要动效？
它的动效是否影响操作速度？
它是否支持多分辨率？
它是否有源文件分层？
它是否能被程序控制？
```

如果这些问题没有答案，就说明它还不是出品级资源。

## 十三、对我现在做 UI 的提醒

我现在最需要改的习惯，是不要太早沉迷最终视觉。

应该先把问题拆清楚：

```text
这个 UI 是什么功能？
玩家最常看哪里？
哪些东西会变化？
哪些东西必须独立成层？
哪些东西要交给程序？
哪些地方需要九宫格？
哪些状态必须补齐？
引擎里实际尺寸是多少？
```

AI、生图、临摹、风格参考都可以帮助我提高美术质量，但真正让 UI 从“像成品”变成“能出品”的，是结构、拆分、适配和测试。

一句话总结：

```text
UI 美术稿解决好不好看。
UI 工程落地解决能不能用。
出品级 UI 必须同时解决这两个问题。
```

## 可以尝试的实践

这周可以做一个小练习：只做一套血条和按钮，不做完整界面。

练习目标：

```text
做一条玩家血条。
做一条 Boss 血条。
做一个主按钮。
做一个技能按钮。
每个都要有拆分资源和引擎测试。
```

血条要求：

```text
外框、空槽、填充、延迟条、高光、低血量特效分开。
测试 100%、50%、5%、0%。
测试中毒和护盾两种状态。
放到亮背景和暗背景上看。
```

按钮要求：

```text
normal、selected、pressed、disabled 四个状态。
测试 2 个字、4 个字、8 个字的按钮文案。
测试鼠标和手柄选中态。
测试点击动效是否足够快。
```

最后输出：

```text
源文件
切图资源
引擎截图
状态演示
问题记录
回修改动
```

如果这个练习能做扎实，比直接做一整套大界面更有价值。
